## вңҰ What is WhiteRoom?

WhiteRoom is a **privacy-first collaborative workspace** built for real-time sessions. Every room is ephemeral, every message is encrypted, and every peer connection is direct вҖ” no data ever passes through a central server in plaintext.

Think Google Meet Г— Notion Г— Miro, but everything is encrypted, offline-capable, and open.

---

## вңҰ Features

### рҹ”җ Security-First Architecture
- **End-to-end encryption** on all data channel messages using AES-GCM symmetric keys
- **RSA key exchange** per peer вҖ” keys are negotiated directly between browsers, never touch the server
- **Waiting room** with host admit/deny вҖ” no one enters without explicit approval
- Zero plaintext content stored server-side

### рҹ“№ Video Conferencing
- Multi-party WebRTC video with adaptive grid layout (1вҖ“6+ participants)
- Per-peer mic and camera toggles with live state sync
- Black track substitution for camera-off (no video track dropped, no reconnect needed)
- TURN server support for NAT traversal

### рҹҺЁ Collaborative Whiteboard
- Powered by **tldraw** вҖ” full-featured drawing, shapes, arrows, text
- Real-time remote cursor overlay with participant names and colors
- Changes broadcast peer-to-peer over encrypted data channels

### рҹ“қ Collaborative Document Editor
- Rich text editing via **TipTap** + **Yjs CRDT** вҖ” true conflict-free real-time collaboration
- Heading, bold, italic, code, blockquote, lists вҖ” full formatting toolbar
- Remote cursor rendering with live caret positions per collaborator
- **Snapshot history** вҖ” save named restore points to the database, restore or delete any time
- Y.UndoManager вҖ” per-user undo/redo that doesn't stomp on collaborators

### рҹӨ– AI Document Summary
- One-click AI summary of the current document via **Groq (Llama 3.3 70B)**
- Streaming response вҖ” tokens appear in real time as they're generated
- Structured output: Overview, Key Points, Action Items, Insights
- Slide-in panel with copy-to-clipboard and regenerate controls

### рҹ’¬ Encrypted Chat
- Per-session ephemeral chat вҖ” messages are encrypted in transit, never persisted
- Deduplication via stable message IDs across peers

### рҹ”„ Offline-First Sync
- **IndexedDB persistence** via `y-indexeddb` вҖ” docs and whiteboard survive refresh and tab close
- Offline banner with automatic state sync on reconnect
- Full Yjs state broadcast to late-joining peers on handshake

### рҹ‘Ҙ Presence & Awareness
- Live presence sidebar with per-user mic/camera status
- Cursor positions broadcast via Yjs Awareness protocol
- Heartbeat awareness updates every 30 seconds

## вңҰ Getting Started

### Prerequisites

- Node.js 18+
- A [Groq API key](https://console.groq.com) (free tier available)
- Optional: TURN server credentials for production NAT traversal

### Installation

```bash
git clone https://github.com/your-username/whiteroom
cd whiteroom
npm install
```

### Environment Variables

Create a `.env.local` file in the project root:

```env
# Required вҖ” AI summarization
GROQ_API_KEY=gsk_xxxxxxxxxxxxxxxxxxxx

# Optional вҖ” TURN server for production WebRTC
TURN_URL=turn:your-turn-server.com:3478
TURN_USERNAME=your_username
TURN_CREDENTIAL=your_credential

# Optional вҖ” database for snapshot persistence
DATABASE_URL=your_database_url
```

### Development

```bash
# Terminal 1 вҖ” Next.js dev server
npm run dev

# Terminal 2 вҖ” Socket.io signaling server
node server/index.js
```

Open [http://localhost:3000](http://localhost:3000).

### Production

```bash
npm run build
npm start
```

> Make sure your signaling server is deployed and `NEXT_PUBLIC_SOCKET_URL` points to it.

### Encryption Flow

```
Peer A opens DataChannel
    в”Ӯ
    в”ңв”Җ Generates AES-GCM room key + RSA keypair
    в”ңв”Җ Sends RSA public key to Peer B
    в”Ӯ
Peer B receives PUBLIC_KEY
    в”ңв”Җ If socket.id < peerId: encrypts AES key with B's RSA public key
    в””в”Җ Sends SET_KEY_SECURE back to A
    в”Ӯ
Both peers now share the same AES-GCM key
All subsequent messages: encrypt(key, payload) вҶ’ decrypt(key, payload)
```

### Offline Sync

Docs and whiteboard state live in a **Yjs document** persisted to IndexedDB. On reconnect or when a new peer joins, the full Yjs state vector is encrypted and sent over the data channel вҖ” both peers converge automatically via CRDT merge.

Chat is intentionally **ephemeral** вҖ” it exists only in React state for the duration of the session.

## вңҰ AI Summary

The AI summary feature reads the current document content directly from the Yjs `tiptap-doc` XML fragment, extracts plain text, and streams a structured summary from **Groq's Llama 3.3 70B** model via SSE.

```
Click "AI Summary"
    в”Ӯ
    в”ңв”Җ AISummaryPanel extracts text from Y.XmlFragment("tiptap-doc")
    в”ңв”Җ POST /api/summarize { docsText }
    в”Ӯ
    в””в”Җ Server streams SSE tokens via Groq SDK
           в”Ӯ
           в–ј
    Panel renders markdown in real time as tokens arrive
```

The summary is structured into four sections when content warrants:
- **рҹ“Ӣ Overview** вҖ” core topic in 1вҖ“2 sentences
- **рҹ”‘ Key Points** вҖ” bullet list of important ideas
- **вң… Action Items** вҖ” tasks and follow-ups
- **рҹ’Ў Insights** вҖ” patterns and open questions

## вңҰ Snapshot System

The document editor supports named snapshots stored in the database. Each snapshot captures the full Yjs document state as a base64-encoded binary blob. Restoring a snapshot applies the state transactionally вҖ” existing content is cleared and replaced cleanly without CRDT merge conflicts.
